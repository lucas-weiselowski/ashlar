Both issues flagged in the original report are fixed on main. gavel's near-total-rewrite case no longer costs tokens — it now falls back to sending the full content whenever a diff would be larger, flooring the worst case at 0% instead of −12.5%. chisel's keyword filter now matches CamelCase exception names (ValueError, ConfigValidationError, etc.) via a second, case-sensitive alternative in LOAD_BEARING_RE, so a traceback's actual exception message survives truncation instead of only its "Traceback" header line. The pooled cut for this run (77.8%) reads lower than the original (84.4%) — that's fixture scale, not regression: this run's synthetic files/logs are sized differently than the originals (see Method), and the two fixes above trade a little compression for correctness in exactly the cases they targeted. Per-scenario deltas below are the meaningful comparison.

Since the last report

FixedGavel no longer costs tokens on heavy rewrites. bin/ashlar now compares the unified diff's estimated size against the raw content before choosing which to emit — if the diff isn't smaller, it sends the full text instead. The ~50%-rewrite scenario that lost −12.5% in the original report now floors at 0.0% (7,922 → 7,922 tokens): still no savings on a heavy rewrite, but never a net loss.
FixedChisel now catches CamelCase exception names. LOAD_BEARING_RE gained a second, case-sensitive alternative (\w*(?-i:Error|Exception)\b) specifically for names like ConfigValidationError that have no word boundary before "Error". Re-running the build-log scenario confirms the exception message line now survives in the output, not just the "Traceback" header — verified directly in this run's chisel output.
Still open: keyword selection is exact, not semantic. Re-running the grep-dump scenario with the same kind of load-bearing-but-unlabeled comment (a circular-dependency warning with no error/warning keyword in it) confirms it's still dropped by head/tail truncation, same as the original report. This is a design tradeoff (fixed keyword list), not a regression — --keep-pattern remains the escape hatch for known-recurring cases.

Method

Each scenario feeds representative content through gavel/chisel at default settings (--max-lines 80), records before/after token counts to the ledger with --record, and reads the total back with ashlar report --by-label. All content here is freshly generated to match the same six shapes as the original report (unchanged/small-diff/large-diff repeat reads; a verbose test run, a large grep dump, a build log with a traceback) — it is not byte-identical to the original fixtures, so absolute before/after counts differ; the per-scenario cut percentages and the two fix confirmations are the numbers to trust, not a strict token-for-token pooled comparison against the original run.

Heuristic, not a real tokenizer. Ashlar estimates tokens as chars / 4 — a stated approximation (bin/ashlar:19-20), not a model-specific tokenizer. Absolute counts here will drift from what a real Claude/GPT tokenizer reports; the relative cut percentages are the trustworthy number.

Findings worth flagging

Gavel is still bimodal by design, just with a floor now. Near-total savings (98.6–99.8%) when a re-read is unchanged or lightly edited; a wash (0.0%) rather than a loss when it's a heavy rewrite. Keying gavel on content that gets substantially rewritten between reads still won't save tokens — it just won't cost them either.
Keyword selection is exact, not semantic (unchanged from the original report). A load-bearing but unlabeled comment mid-list still gets cut by head/tail truncation. This is inherent to a fixed keyword list, not something either fix addressed — --keep-pattern is the mitigation, per scenario or recurring term.

Recommendations

  • Gavel is now safe as a default for any repeat-read key, not just ones expected to stay mostly unchanged — the diff-size floor guarantees it never costs more than sending the content plain. It's still most effective on keys that are re-read mostly unchanged (file watches, config re-checks, polling loops); heavy-rewrite keys just see it break even instead of losing tokens.
  • No further action needed on the CamelCase gap — confirmed fixed and held under a fresh traceback fixture in this run.
  • Chisel remains a safe default filter for high-volume, low-signal output (test runs, build logs, grep dumps): 88.8–98.5% cut held across three different shapes of noisy output in this run.
  • For load-bearing lines that don't use error/warning language (e.g. a circular-dependency note, a perf regression comment), reach for --keep-pattern rather than expecting the default keyword list to catch them.
