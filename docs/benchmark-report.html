Chisel is the reliable cut — 94.6–99.3% across a verbose test run, a large grep dump, and a build log with an embedded stack trace. Gavel is bimodal — near-total savings (98.7–99.6%) when a re-read is unchanged or lightly edited, but a net loss (−12.5%) when the re-read is a near-total rewrite, because a unified diff of two mostly-different files is larger than either file alone. Pooled across everything tested, 1 in every ~6.3 input tokens survives compaction — roughly 84% is removed before it would otherwise reach the model.

Method

Each scenario feeds real or representative content through gavel/chisel at default settings (--context 2, and --max-lines set per scenario to match a realistic viewport), records before/after token counts to the ledger with --record, and reads the total back with ashlar report --by-label. Two scenarios use real content generated on this machine (the ashlar CLI's own source, this repo's pytest run); four use synthetic-but-representative content built to match a named, common agent tool-output pattern (grep dump, CI test run, build log with a traceback) — each is labeled as such in the table below.

Heuristic, not a real tokenizer. Ashlar estimates tokens as chars / 4 — a stated approximation (bin/ashlar:19-20), not a model-specific tokenizer. Absolute counts here will drift from what a real Claude/GPT tokenizer reports; the relative cut percentages are the trustworthy number.

Findings worth flagging

Gavel can cost tokens on heavy rewrites. A unified diff of two versions that share little content is larger than either version alone — confirmed at −12.5% on a ~50%-rewritten file. Gavel is a net win only when repeat reads are usually unchanged or lightly edited (the common case for re-reading a file mid-edit-loop), not when the same key is reused across unrelated content.
Chisel's keyword filter misses CamelCase exception names. The load-bearing regex is \berror\b (word-boundary anchored) — it correctly matches "Traceback" and standalone "error," but not ValueError, KeyError, TypeError, or any SomethingError class name, because there's no word boundary between the preceding letter and "Error." In the build-log scenario, the traceback's header line survived (via the "Traceback" keyword) but the actual exception message — the part that says what broke — fell outside the default 2-line context window and was elided. Verified directly against bin/ashlar:25-29.
Keyword selection is exact, not semantic. In the grep-dump scenario, a single comment flagging a real circular-dependency risk sat mid-list with no matching keyword and was dropped by head/tail truncation along with the surrounding noise. Chisel reduces to what looks like an error/warning; a load-bearing line that isn't phrased that way can still be cut.

Recommendations

  • Use gavel for keys you expect to be re-read mostly unchanged — file watches, config re-checks, polling loops. Don't key gavel on content that gets substantially rewritten between reads; there the diff overhead exceeds the savings.
  • Widen the CamelCase gap before relying on chisel for uncaught-exception triage. Extending LOAD_BEARING_RE to catch e.g. \w*Error\b/\w*Exception\b alongside the existing terms would keep the actual exception message in frame, not just the "Traceback" header.
  • Chisel is safe as a default filter for high-volume, low-signal output (test runs, build logs, grep dumps) where the goal is finding a needle, not preserving every line — 94.6–99.3% cut held across three very different shapes of noisy output.

Reproduce this

All six runs are logged under bench:* labels in ~/.ashlar/ledger.jsonl:

$ ashlar report --by-label Label Count Rough Perfect Saved % bench:chisel-stacktrace-log 1 9,215 61 9,154 99% bench:chisel-grep-dump 1 7,912 426 7,486 95% bench:chisel-verbose-testrun 1 5,807 172 5,635 97% bench:gavel-unchanged-repeat 1 2,453 9 2,444 100% bench:gavel-small-diff 1 2,456 31 2,425 99% bench:gavel-large-diff 1 3,747 4,214 -467 -12%
